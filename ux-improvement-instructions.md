# UX改善作業指示書（arent-test-v3）

## 前提条件
- `docker compose up --build` で全4サービスが正常稼働していること
- Step 1〜6（R1-R5修正）が完了済みであること
- フロントエンド: `frontend/` ディレクトリ（Next.js + TypeScript + Tailwind CSS）

## 改善概要

フロントエンド目視確認で検出された4つのUX課題を修正する。
機能は全て動作済みのため、**視認性・フィードバック・ナビゲーション**の改善に集中する。

| # | 課題 | 優先度 | 影響範囲 |
|---|------|--------|---------|
| UX-1 | 状態遷移のNG/OK判断が視覚的に不明確 | HIGH | 指摘一覧・詳細パネル |
| UX-2 | 対応箇所（指摘ピン）がわかりにくい | HIGH | 3Dビューア・指摘一覧連携 |
| UX-3 | 写真追加後に何をしたかわからない | MEDIUM | 写真アップロード・詳細パネル |
| UX-4 | 写真の保存先が不明 | LOW | 写真サムネイル |

## Step 1: 状態遷移の視認性強化（UX-1 解消）

### 1.1 問題の詳細

指摘の状態バッジ（未対応/対応中/完了）が小さなテキストラベルのみで表現されており、
どの遷移が許可されていてどれがNGなのかが視覚的に判別できない。
現状: 赤い「未対応」バッジと緑の「完了」バッジは色で区別できるが、
遷移操作時のフィードバックが不足している。

**関連ファイル（推定）**:
- `frontend/src/components/` 配下の指摘一覧・詳細コンポーネント
- 状態バッジコンポーネント

### 1.2 修正

以下の3点を実装する:

**A) 状態バッジのCSS強化**

```
状態バッジに以下のスタイルを追加:
- 未対応: 赤背景 + 左ボーダー4px赤 + アイコン（⚠ または ●）
- 対応中: 黄/オレンジ背景 + 左ボーダー4px黄 + パルスアニメーション（現在作業中を示す）
- 完了: 緑背景 + 左ボーダー4px緑 + チェックマークアイコン（✓）
- 差し戻し: グレー背景 + 左ボーダー4pxグレー + 戻り矢印アイコン
```

**B) 状態遷移ボタンの許可/不許可表示**

```
指摘詳細パネルに状態遷移ボタンを配置:
- 許可された遷移: 通常ボタン（クリック可能・色付き）
- 不許可の遷移: disabled + グレーアウト + ツールチップ「この状態からは遷移できません」
- 遷移成功時: ボタンにチェックマークを0.5秒表示後、新状態のバッジに切り替え

遷移ルール（ドメイン層に準拠）:
  未対応 → 対応中 / 完了
  対応中 → 完了 / 差し戻し（未対応に戻す）
  完了 → 差し戻し（未対応に戻す）
  ※ ドメイン層のIssue状態遷移ロジックを確認し、上記と異なる場合はドメイン層に従うこと
```

**C) 一覧での状態カラムにインジケーター追加**

```
指摘一覧の各行に:
- 左端に状態色のドット（●）を常時表示（テーブル表示の場合）
- または カード表示の場合は左ボーダーを状態色に設定
- 選択中の指摘行にはリング状のアウトライン（focus-visible相当）を追加
```

### 1.3 検証

```bash
# フロントエンドのビルドが通ることを確認
cd frontend && npm run build

# 期待: ビルド成功・警告なし
```

**ブラウザ目視確認**:
1. http://localhost:3000 で指摘一覧を表示 → 各指摘の左端に状態色ドットが表示されること
2. 指摘詳細を開く → 状態遷移ボタンが表示され、不許可遷移がdisabledであること
3. 「未対応」→「対応中」に遷移 → ボタンにチェックマーク表示後、バッジが黄色に変わること
4. 「完了」状態の指摘で「対応中」ボタンがdisabledであること（直接遷移不可の場合）

### 1.4 ゲート（次Stepへの進行条件）

- `npm run build` が成功すること
- 3種類以上の状態バッジが視覚的に区別可能であること（色＋アイコン）
- 不許可遷移がdisabledで表示されること

## Step 2: 指摘ピンのハイライト・対応箇所の視認性強化（UX-2 解消）

### 2.1 問題の詳細

3Dビューア上の指摘ピン（マーカー）が、どの指摘に対応しているのかわかりにくい。
一覧で選択した指摘と3Dビューアのピンの連動が視覚的に不十分。
ステータスバーに「クリック: 部材選択 / ダブルクリック: 空間指摘」と表示されているが、
登録済みの指摘ピンがどこにあるのかが視覚的に埋もれている。

**関連ファイル（推定）**:
- APS Viewer連携コンポーネント（Three.js / Forge Viewer Extension）
- 指摘ピン描画ロジック

### 2.2 修正

**A) 指摘ピンの状態色連動**

```
3Dビューア上の指摘ピン（マーカー）の色を状態に連動させる:
- 未対応: 赤ピン
- 対応中: 黄/オレンジピン
- 完了: 緑ピン

ピンのサイズ:
- 通常: 12px
- ホバー: 16px + ツールチップ（指摘タイトル表示）
- 選択中: 20px + パルスアニメーション（波紋エフェクト）
```

**B) 一覧 ↔ ピンの双方向ハイライト**

```
1. 一覧で指摘をクリック → 対応する3Dピンにカメラをズーム＋パルスアニメーション
2. 3Dピンをクリック → 一覧の対応する指摘行をハイライト＋スクロールして表示

ハイライトのスタイル:
- 一覧側: 背景色を状態色の薄い版（opacity: 0.1）に変更 + 左ボーダー強調
- 3Dピン側: リング状のパルスアニメーション（2秒で減衰）
```

**C) 新規指摘登録時のピン配置フィードバック**

```
ピン登録モードで位置をクリックした際:
- クリック位置にプレビューピン（点線の円）を即座に表示
- 登録フォーム上部に「位置: Element | dbId: XXXX | (X, Y, Z)」を表示（現状実装済み）
- 登録完了後: プレビューピンが確定ピン（実線）にアニメーション遷移
```

### 2.3 検証

```bash
cd frontend && npm run build
```

**ブラウザ目視確認**:
1. 指摘一覧の任意の項目をクリック → 3Dビューアのカメラが対応ピンにズームすること
2. 3Dビューアのピンをクリック → 一覧側の該当行がハイライトされること
3. 各ピンの色が指摘の状態（未対応=赤/完了=緑）に対応していること
4. ピンにホバー → 指摘タイトルのツールチップが表示されること

### 2.4 ゲート（次Stepへの進行条件）

- 一覧クリック → ピンハイライトの連動が動作すること
- ピンの色が状態に対応していること（最低2色以上の区別）
- `npm run build` が成功すること

## Step 3: 写真アップロードのフィードバック強化（UX-3 解消）

### 3.1 問題の詳細

写真をアップロードした後、ユーザーに「何が起きたか」のフィードバックがない。
サムネイルが詳細パネルに表示されるが、追加されたことを示す視覚的な合図が不足。
是正前/是正後の区別はラベルで表示されているが、追加タイミングが不明。

**関連ファイル（推定）**:
- 写真アップロードコンポーネント
- 指摘詳細パネルコンポーネント

### 3.2 修正

**A) アップロード完了トースト通知**

```
写真アップロード成功時:
- 画面右上にトースト通知「写真をアップロードしました」（3秒で自動消去）
- トーストにサムネイルプレビュー（小）を含める
- エラー時: 赤トースト「アップロードに失敗しました。再試行してください」

トースト実装:
- 既存のUIライブラリにトーストがあればそれを使用
- なければ、シンプルなトーストコンポーネントを作成（position: fixed, top-right, z-index: 50）
```

**B) 新規追加写真のハイライト**

```
写真追加直後のサムネイルに:
- 「NEW」バッジを3秒間表示
- ボーダーを青色でパルスアニメーション（1回）
- 写真一覧の先頭にスクロール（新しい写真が上に来る場合）
```

**C) 写真カウントのリアルタイム更新**

```
指摘一覧の各行に表示されている写真アイコン+数字（📷 1）を:
- アップロード完了時に即座にカウントアップ
- カウント変更時に数字を一瞬拡大（scale: 1.2 → 1.0, 0.3秒）
```

### 3.3 検証

```bash
cd frontend && npm run build
```

**ブラウザ目視確認**:
1. 指摘詳細で「写真追加」→ ファイル選択 → アップロード完了後にトースト通知が表示されること
2. 新規追加された写真サムネイルに「NEW」バッジが一時表示されること
3. 一覧の写真カウントが即座に更新されること
4. アップロードエラー時（ネットワーク切断等）に赤トーストが表示されること

### 3.4 ゲート（次Stepへの進行条件）

- アップロード完了時にトースト通知が表示されること
- 写真カウントがリアルタイムで更新されること
- `npm run build` が成功すること

## Step 4: 写真保存先の可視化（UX-4 解消）

### 4.1 問題の詳細

写真がどこに保存されているのか（MinIOストレージ）がUIから判別できない。
Presigned URLは `localhost:9000` ベースで正しく生成されているが、
ユーザーにはストレージの所在が不透明。

### 4.2 修正

**A) 写真サムネイルのホバー情報**

```
写真サムネイルにホバーした際にツールチップを表示:
- 「ストレージ: MinIO (localhost:9000)」
- 「アップロード日時: YYYY-MM-DD HH:MM」
- 「ファイルサイズ: XX KB」（バックエンドAPIから取得可能な場合）

※ 本番環境ではストレージ情報の表示は不要かもしれないが、
  PM業務体験課題としては「アーキテクチャ理解を示す」ために有用
```

**B) 写真の拡大表示（ライトボックス）**

```
サムネイルクリック時:
- オーバーレイで拡大表示（モーダル or ライトボックス）
- 拡大画像の下部にメタ情報（是正前/是正後・アップロード日時）
- ESCまたはオーバーレイクリックで閉じる
- 左右矢印キーで同一指摘内の写真を切り替え（複数写真がある場合）
```

### 4.3 検証

```bash
cd frontend && npm run build
```

**ブラウザ目視確認**:
1. 写真サムネイルにホバー → ツールチップにストレージ情報が表示されること
2. サムネイルクリック → ライトボックスで拡大表示されること
3. ESCキーでライトボックスが閉じること
4. 拡大画像の下部にメタ情報が表示されること

### 4.4 ゲート（次Stepへの進行条件）

- サムネイルホバーでツールチップが表示されること
- ライトボックスの開閉が正常動作すること
- `npm run build` が成功すること
